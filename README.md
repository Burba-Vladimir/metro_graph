# Moscow Metro — Force-Directed Graph

> If you ever wanted to see Moscow Metro the way physics intended — 300+ stations
> finding their natural positions through attraction and repulsion — here's your chance.

A D3 v7 force-directed visualization of the Moscow Metro network. Stations are nodes,
line connections are edges, and transfer hubs pull stations on different lines together.
The three ring lines (Кольцевая, БКЛ, МЦК) are constrained to perfect draggable circles
whose radii emerge dynamically from the simulation.

## Quick start

Just open `metro_graph.html` in any modern browser — no server, no dependencies, everything is bundled.

## Controls

| Action | How |
|---|---|
| Pan | Drag the background |
| Zoom | Scroll wheel or ＋ / − buttons |
| Rotate | ⟲ / ⟳ buttons (15° steps); label shows current angle |
| Reset view | **H** button |
| Drag a station | Click and drag a node |

## Building from source

Run both scripts from the **repository root**:

```bash
python src/build_graph.py   # data/metro_data.json  →  data/metro_graph.json
python src/build_html.py    # data/metro_graph.json + src/metro_vis.js + vendor/d3.min.js  →  metro_graph.html
```

Requires Python 3.8+ with no third-party libraries.

## Editing the data

All topology lives in **`data/metro_data.json`**. The file has two top-level keys:

### `stations`

An array of station objects scraped from Wikipedia and hand-corrected:

```json
{
  "code": "068",
  "name": "Киевская",
  "lineName": "Филёвская линия",
  "lineColor": "#009EE0",
  "date": "1958-03-05",
  "transfers": [
    "Переход на станцию «Киевская» Арбатско-Покровской линии",
    "Переход на станцию «Киевская» Кольцевой линии"
  ]
}
```

### `lines`

Each line entry defines its topology through one or more **segments** — ordered lists of station names that the build script connects sequentially:

```json
{
  "name": "Филёвская линия",
  "color": "#009EE0",
  "prefix": "L4",
  "ring": false,
  "segments": [
    ["Кунцевская", "Пионерская", "...", "Киевская", "...", "Александровский сад"],
    ["Киевская", "Деловой центр", "Москва-Сити"]
  ]
}
```

Branched lines (Филёвская is the only one) use multiple segments that share a branch-point station name. Ring lines use `"ring": true` and a single segment; the build script adds the wrap-around edge automatically.

To fix an incorrect transfer, edit the `transfers` array of the relevant station directly and rebuild.

## How it works

`build_graph.py` reads the station list and line topology from `metro_data.json` and
produces a graph with three edge types:

- **line** — sequential connections along each line segment
- **transfer** — short links between same-named stations on different lines (parsed from the `transfers` field)

`metro_vis.js` runs a D3 force simulation on that graph. After each physics tick, ring nodes
are projected onto a perfect circle whose radius is derived from the average chord length
between adjacent ring stations, smoothed with an exponential moving average. This keeps
the rings circular while letting the rest of the graph settle naturally.
